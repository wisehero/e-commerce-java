# 작업 방식

모든 세션·모든 작업에서 항상 적용. AGENTS.md 또는 CLAUDE.md를 읽으면 가장 먼저 이 파일을 함께 읽는다.

## 1. 직접 작성한다

프로덕션 코드를 직접 작성·수정한다. (구 규칙 "작성하지 말고 제안하라"는 **폐기됨** — 2026-06-13.)

- 프로덕션 코드를 사용 가능한 편집 도구(예: patch 적용)로 직접 변경한다. 스니펫만 던지고 사용자 적용을 기다리지 않는다.
- 단, 변경하면서 **의도와 트레이드오프는 계속 설명한다** (무엇을·왜 바꿨는지, 영향 범위). DDD 5계층 규칙(`ddd.md`)은 그대로 준수한다.
- 되돌리기 어렵거나 외부로 나가는 작업(force push, 배포, 외부 전송, 대량 삭제 등)은 먼저 확인을 받는다.
- 테스트(`src/test/...`, `src/testFixtures/...`)·`wiki/`·`.agents/`·`.claude/rules/` 메타 파일도 직접 작성·수정한다.
- `.claude/skills/`는 외부/개인 스킬 자산으로 취급해 수정하지 않는다. 스킬 문서와 프로젝트 문서가 어긋나도 프로젝트 문서(`AGENTS.md`, `CLAUDE.md`, `.agents/rules/`, `.claude/rules/`, `wiki/`) 쪽에서만 보정한다.

### 변경 설명 — 무엇을·왜 함께 보여라

코드를 바꾼 뒤(또는 바꾸면서) 다음을 함께 설명한다.

- 대상 파일 경로 (있다면 라인 번호).
- 추가/변경/삭제한 핵심.
- 변경 의도와 영향 범위.
- 대안이나 트레이드오프가 있다면 함께.

## 2. 작업 시작 전 관련 규칙을 읽어라

도메인별 규칙은 별도 파일에 있다. 매번 읽고 시작하라.

- Java 코드 작업 전 `.agents/rules/ddd.md`를 먼저 읽어라.
- 커밋 작업은 `.agents/rules/commit.md` 규칙을 따른다 (`/commit` 명령으로 자동 적용).

## 3. 코딩 전에 생각하라

추측하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.

제안하기 전에:

- 가정을 명시적으로 밝혀라. 확실하지 않으면 질문하라.
- 여러 해석이 가능하면 모두 제시하라 — 조용히 하나를 고르지 마라.
- 더 단순한 접근법이 있다면 말하라. 필요하면 반론을 제기하라.
- 뭔가 불명확하면 멈춰라. 무엇이 헷갈리는지 짚어라. 질문하라.
- 읽기 우선: 제안 대상 코드를 먼저 읽어라. 읽지 않고 제안하지 마라.
- 환각 금지: 존재하지 않는 API, 패키지, 파일 경로, 설정 옵션을 지어내지 마라. 확실하지 않으면 먼저 확인하라.

## 4. 단순함 우선

문제를 해결하는 최소한의 코드. 짐작으로 쓴 코드는 금지.

- 요청받지 않은 기능을 추가하지 마라.
- 한 번만 쓰이는 코드에 추상화를 만들지 마라.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 마라.
- 발생할 수 없는 시나리오에 대한 에러 핸들링을 하지 마라.
- 200줄로 제안했는데 50줄로 가능하면 다시 작성하라.
- 자문하라: "시니어 엔지니어가 이거 오버엔지니어링이라고 할까?" 그렇다면 단순화하라.

## 5. 목표 중심 제안

성공 기준을 정의하라. 제안과 함께 어떻게 검증할지를 명시하라.

요청을 검증 가능한 목표로 변환하라:

- "유효성 검사 추가" → "잘못된 입력에 대한 테스트로 검증. 제안에 테스트 코드 포함."
- "버그 수정" → "재현 테스트로 검증. 제안에 재현 시나리오와 수정 코드 포함."
- "X 리팩토링" → "리팩토링 전후로 테스트 통과 확인. 사용자가 적용 후 `./gradlew test`."

다단계 제안에는 간략한 계획을 제시하라:

```
1. [제안 단계] → 검증: [사용자가 확인할 방법]
2. [제안 단계] → 검증: [사용자가 확인할 방법]
```

## 6. 한 번에 하나씩 변경하라

논리적으로 독립된 변경은 하나씩 끝내고 검증한 뒤 다음으로 간다. 무관한 변경을 한 덩어리로 섞지 마라.

- 깨졌을 때 원인을 특정할 수 있도록 변경 단위를 작게 유지하라.
- 사용자가 "전부 해"처럼 묶음 진행을 지시하면 관련 작업을 이어서 진행하되, 각 단계 후 검증(빌드/테스트)으로 확인하라.
- 설계 결정(API 계약·되돌리기 어려운 선택)에서 합리적 기본값이 없고 트레이드오프가 크면, 진행 전에 짚고 결정을 받아라.
- **예외**: 한 변경이 여러 파일에 강하게 묶여 있어 분리하면 작동하지 않는 경우 묶어 처리하고 그 이유를 명시하라.

## 7. 실패 대응

에러가 나면 원인부터 파악하라. 같은 시도를 반복하지 마라.

- 에러 발생 시 증상만 고치지 말고 근본 원인을 분석하라.
- 같은 명령을 재시도하기 전에 왜 실패했는지 먼저 이해하라.
- 방향이 틀렸으면 고치려 하지 말고 과감히 버리고 다시 제안하라. 매몰 비용에 집착하지 마라.

## 8. PR 생성 기본값

GitHub PR을 올릴 때는 사용자가 명시적으로 "draft" 또는 "초안"을 요청한 경우에만 Draft PR로 만든다.

- "PR 올려", "PR 생성", "커밋과 PR 생성"은 기본적으로 **Ready for review PR**을 의미한다.
- `gh pr create`를 사용할 때 `--draft`를 붙이지 않는다.
- GitHub 도구의 `draft` 옵션은 기본 `false`로 둔다.
- PR 생성 직후 `isDraft=false`인지 확인한다.
- 실수로 Draft PR이 만들어졌다면 즉시 ready-for-review로 전환하고 다시 확인한다.

## 9. 사용자 질문 도구 사용 규칙

닫힌 질문에는 현재 에이전트 환경에서 사용자 질문 도구가 제공될 때만 그 도구를 사용하라. 열린 질문이나 질문 도구가 없는 환경에서는 텍스트로 질문하고 사용자의 다음 메시지를 기다려라.

### 도구 선택

- **닫힌 질문** (선택지가 명확할 때) → 사용 가능한 사용자 질문 도구 사용
  - 예: "이어서 진행" / "새로 시작", "빌드 건너뛰기" / "직접 입력"
- **열린 질문** (자유 입력이 필요할 때) → 질문을 텍스트로 출력하고 사용자의 다음 메시지를 기다린다
  - 예: "이 도메인에서 풀려는 문제가 뭔가?", "관련 레포가 있는가?"

### 한 번에 한 가지만 물어라

질문을 묶어 던지지 마라. 사용자가 결정 하나에 집중할 수 있게 하라.

- 코드 제안 후 검토 포인트나 설계 결정에 대한 질문이 여러 개라도 가장 중요한 것 하나만 먼저 묻고, 답을 받은 뒤 다음으로 넘어가라.
- 결정들이 강하게 묶여 있어 동시에 결정해야 하는 경우만 예외로 묶어 묻고, 그 이유를 명시하라.

### 빈 응답 처리

- 사용자 질문 도구에서 빈 응답(선택 없음, 텍스트 없음)을 받으면 동의/승인/건너뛰기로 간주하지 마라.
- 빈 응답 시 1회 재질문한다.
- 2회 연속 빈 응답이면 작업을 중단하고 사용자에게 상황을 설명한다.

## 10. 피드백 반영

사용자가 실수를 지적하면 기록하라. 작업 중 개선점을 발견하면 제안하라.

- 같은 세션에서 지적받은 실수는 즉시 반영하고, 이후 동일 실수를 반복하지 마라.
- 사용자가 내 결과물을 직접 수정했다면, 수정 의도를 분석하여 컨벤션으로 제안하라.
- 코드 탐색 중 wiki/에 없는 도메인 지식을 발견하면 문서화를 제안하라.
- **사용자의 기술 질문(개념·설계·트레이드오프·언어/프레임워크 사용법)에 답변한 후, 핵심 내용을 `wiki/learning/`에 정리해 보관하도록 제안하라.** 같은 질문이 반복되지 않게 학습 자산으로 축적한다.
- 같은 유형의 요청이 반복되면 skill 생성 또는 기존 skill 확장을 제안하라.
- 작업 흐름에서 수동/반복 작업을 발견하면 스크립트나 skill로 자동화를 제안하라.

### 발견 내용별 반영 위치

발견 내용에 따라 적절한 위치에 반영을 제안하라.

| 발견 내용 | 반영 위치 |
|---|---|
| 워크스페이스 전체 규칙 | `AGENTS.md` 또는 `.agents/rules/` |
| 비즈니스 정책 / 의사결정 | `wiki/{도메인}/` |
| 사용자 기술 질문 답변 | `wiki/learning/` |
