# Matt Pocock Skills (프로젝트 전용 복사본)

[mattpocock/skills](https://github.com/mattpocock/skills) 플러그인의 공식 14개 스킬을 이 프로젝트 전용으로 복사해 둔 것이다. 글로벌 플러그인이 아니라 `.claude/skills/` 로컬 스킬로 둬서 다른 프로젝트엔 영향을 주지 않는다.

각 스킬을 **언제** 발동해야 하는지를 정리한다. 상세 동작은 각 디렉토리의 `SKILL.md`를 참고.

---

## 첫 실행 전 — 셋업 필수

### setup-matt-pocock-skills
**무엇**: 이 레포의 이슈 트래커(GitHub / 로컬 마크다운), triage 라벨 어휘, 도메인 문서 위치를 `AGENTS.md`/`CLAUDE.md`와 `docs/agents/`에 기록한다.

**언제**:
- `to-issues`, `to-prd`, `triage`, `diagnose`, `tdd`, `improve-codebase-architecture`, `zoom-out` 을 **처음 쓰기 전**.
- 위 스킬들이 이슈 트래커·라벨·도메인 문서를 못 찾고 헤맬 때.

> 이 스킬은 `disable-model-invocation: true` — 모델이 자동으로 부르지 않는다. 사용자가 직접 `/setup-matt-pocock-skills`로 실행해야 한다.

## Engineering — 코드 작업용 (10개)

### diagnose
**무엇**: 어려운 버그·성능 회귀 진단 루프. 재현 → 최소화 → 가설 → 계측 → 수정 → 회귀 테스트.

**언제**:
- "이거 진단해줘 / 디버깅해줘 / 왜 깨졌지" 같은 요청.
- 버그 리포트, "뭐가 throw 됨 / 실패함 / 안 됨" 형태의 보고.
- 성능 회귀(느려졌다, 메모리 늘었다) 보고.

### tdd
**무엇**: red-green-refactor TDD 루프. 수직 슬라이스 단위로 기능을 짓거나 버그를 고친다.

**언제**:
- TDD로 새 기능을 만들고 싶다고 명시할 때.
- "red-green-refactor", "통합 테스트로 시작", "테스트부터 짜자" 같은 표현.
- 버그 픽스를 테스트 우선으로 잡고 싶을 때.

### grill-with-docs
**무엇**: 작성 중인 계획을 도메인 모델·기존 문서(`CONTEXT.md`, ADR)에 비춰 무자비하게 인터뷰하면서, 결정이 굳어질 때마다 문서를 인라인으로 업데이트한다.

**언제**:
- 변경을 시작하기 전 계획을 프로젝트의 언어·결정과 정렬하고 싶을 때.
- "이 설계 검증해줘", "지금 계획이 도메인 용어에 맞나" 같은 요청.
- `grill-me`보다 더 강하게, 코드/문서 컨텍스트까지 끌어와서 인터뷰받고 싶을 때.

### improve-codebase-architecture
**무엇**: `CONTEXT.md`의 도메인 언어와 `docs/adr/`의 결정에 비춰 코드베이스의 deepening(깊이) 기회를 찾는다 — 결합도 높은 모듈 정리, 테스트 용이성·AI 탐색성 개선.

**언제**:
- "아키텍처 개선해줘 / 리팩토링 기회 찾아줘 / 더 깊게 만들자".
- 도메인 결합이 의심스러운 모듈을 진단할 때.
- 큰 PR을 앞두고 구조부터 손볼지 결정할 때.

### prototype
**무엇**: 본 작업에 들어가기 전 일회용 프로토타입을 만들어 설계를 살에 붙여본다. 두 분기 — (1) 상태/비즈니스 로직 검증용 실행 가능한 터미널 앱, (2) 한 라우트에서 토글되는 여러 UI 변형.

**언제**:
- "프로토타입 만들어줘 / 한번 갖고 놀아보자 / 디자인 몇 가지 시도".
- 데이터 모델·상태 머신이 정말 맞는지 손으로 확인하고 싶을 때.
- UI 방향을 정하기 전 여러 안을 옆에 두고 비교하고 싶을 때.

### triage
**무엇**: 트리아지 역할 기반의 상태 머신으로 이슈를 분류·정리한다.

**언제**:
- 새 이슈를 만들 때.
- 들어온 버그 리포트·기능 요청을 정리할 때.
- AFK 에이전트(자동 작업자)에게 넘길 이슈를 다듬을 때.
- 이슈 워크플로우 자체를 정돈하고 싶을 때.

### to-issues
**무엇**: 계획·스펙·PRD를 tracer-bullet 수직 슬라이스로 잘라 독립적으로 잡을 수 있는 이슈로 분해한다.

**언제**:
- 합의된 계획을 이슈 보드로 옮길 때.
- 큰 작업을 누가 봐도 하나씩 잡을 수 있는 티켓으로 쪼개고 싶을 때.

### to-prd
**무엇**: 현재 대화 컨텍스트를 PRD 형식으로 정리해 이슈 트래커에 등록한다.

**언제**:
- 즉흥 논의가 충분히 익어 PRD로 굳히고 싶을 때.
- "지금까지 얘기한 거 PRD로 만들어줘" 요청.

### zoom-out
**무엇**: 좁아진 시야에서 강제로 줌아웃해 더 큰 맥락·고수준 관점을 끌어온다.

**언제**:
- 처음 보는 영역의 코드 앞에서 막혔을 때.
- 작은 변경에 매몰돼 전체 그림을 잃었다고 느낄 때.
- "이게 큰 그림에서 어디 붙는 거지" 같은 질문.

> `disable-model-invocation: true` — 명시적으로 호출해서 쓴다.

## Productivity — 워크플로우 보조 (4개)

### grill-me
**무엇**: 사용자의 계획·설계를 의사결정 트리의 모든 분기가 풀릴 때까지 인터뷰한다(코드 컨텍스트 없이).

**언제**:
- "이 계획 grill 해줘 / 무자비하게 따져줘 / 빠진 거 짚어줘".
- 비코드 결정(스펙·기획·운영)을 압박 테스트하고 싶을 때.
- 코드까지 끌어오는 건 과해서 `grill-with-docs`는 무거울 때.

### caveman
**무엇**: 초압축 대화 모드. 군더더기·관사·인사말을 다 잘라 토큰 ~75% 절약. 기술 정확성은 유지.

**언제**:
- "caveman 모드 / 짧게만 / 토큰 아껴".
- 긴 세션에서 컨텍스트 윈도우가 빠듯할 때.
- 단순 빠른 응답만 필요한 구간.

### handoff
**무엇**: 현재 대화를 다른 에이전트(또는 새 세션)가 이어받을 수 있게 핸드오프 문서로 압축한다.

**언제**:
- 세션이 길어져 컨텍스트가 무거워질 때 새 세션으로 갈아타기 직전.
- 다른 사람·다른 에이전트에게 작업을 넘길 때.
- 작업 중단 직전 상태를 남겨두고 싶을 때.

### write-a-skill
**무엇**: 새 스킬을 적절한 구조·점진적 disclosure·번들 자원과 함께 작성한다.

**언제**:
- 새 스킬을 만들고 싶을 때.
- 반복되는 워크플로우를 스킬로 추출하고 싶을 때.
- 기존 스킬을 본떠 변형판을 만들 때.

## 이 프로젝트와의 충돌점 — 발동 전 확인

1. **`behavior.md`의 "제안만, 직접 수정 금지" 원칙**과 일부 스킬(특히 `tdd`, `prototype`, `diagnose`)이 충돌한다. 이 스킬들은 에이전트가 직접 코드를 쓰는 흐름을 가정한다.
   → 발동 시 `behavior.md` 우선. 테스트 코드(`src/test/`, `src/testFixtures/`)는 직접 쓸 수 있지만 프로덕션 코드는 채팅에 스니펫으로 제안.

2. **Matt 본인 컨벤션 참조**: `grill-with-docs`, `improve-codebase-architecture` 가 `CONTEXT.md`, `docs/adr/` 를 가리킨다. 이 프로젝트는 `wiki/` 구조이므로 `setup-matt-pocock-skills` 실행 시 매핑을 명시한다.

3. **응답 언어**: 스킬 본문은 영어다. 응답은 한국어로 유지해야 한다 (`behavior.md` 흐름과 wiki의 한국어 톤에 맞춤).

4. **DDD 5계층 규칙**: `tdd`·`prototype`이 만들 코드가 `.claude/rules/ddd.md`의 의존성 방향·계층 어휘를 깨지 않도록 발동 전 ddd.md를 함께 읽힌다.
