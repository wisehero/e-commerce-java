# 장바구니 정책

작성: 2026-06-13 · 상태: 초안

이 문서는 장바구니가 고객의 구매 의도를 어떻게 보관하고, 상품·SKU 상태 변화에 어떻게 반응하는지를 정의한다.
구현 구조와 기술적 근거는 [`cart-domain-spec.md`](./cart-domain-spec.md)를 따른다.

## 1. 목적

장바구니 정책은 고객이 구매 후보를 담고 조정하는 과정에서 지켜야 할 비즈니스 규칙을 정의한다.

이 정책은 다음 질문에 답한다.

- 장바구니는 어떤 정보를 저장하는가?
- 장바구니 상품 정보는 담은 시점 기준인가, 현재 시점 기준인가?
- 상품이 품절·판매중지·단종되면 장바구니에는 어떻게 보이는가?
- 같은 SKU를 다시 담으면 어떻게 처리하는가?
- 장바구니와 주문은 어떻게 연결되는가?

## 2. 핵심 용어

| 용어 | 의미 |
|---|---|
| 장바구니 | 회원이 구매 후보 SKU와 수량을 임시로 보관하는 공간 |
| 장바구니 라인 | 장바구니에 담긴 SKU와 수량 |
| live 참조 | 담은 시점 정보가 아니라 조회 시점의 상품·SKU 정보를 다시 읽는 방식 |
| 구매 가능 라인 | 현재 상품 상태, 브랜드 상태, 재고 기준으로 주문 가능한 장바구니 라인 |
| 구매 불가 라인 | 판매중지·단종·브랜드 비활성·상품/SKU 없음 등으로 주문할 수 없는 라인 |
| 품절 라인 | 상품은 판매중이지만 현재 SKU 재고가 부족한 라인 |

## 3. 소유자 정책

장바구니는 회원을 기준으로 소유된다.

- 장바구니 소유자는 회원 ID다.
- 회원 한 명은 하나의 장바구니를 가진다.
- 회원이 처음 상품을 담을 때 장바구니를 만든다.
- 존재하지 않는 회원의 장바구니는 만들 수 없다.
- 장바구니가 없는 회원의 장바구니 조회는 실패가 아니라 빈 장바구니로 응답한다.
- 현재는 인증이 없으므로 회원 본인 여부를 강제하지 않는다.

게스트 장바구니는 현재 정책 범위가 아니다.

## 4. 저장 정책

장바구니는 구매 후보의 최소 정보만 저장한다.

- 장바구니 라인은 SKU ID와 수량만 저장한다.
- 상품명, 옵션, 가격, 재고, 판매 상태는 저장하지 않는다.
- 장바구니 총액도 저장하지 않는다.
- 장바구니 총액은 조회 시점의 현재 판매가로 계산한다.

이 정책은 장바구니가 과거 구매 사실이 아니라 현재 구매 의도라는 점을 반영한다.
과거 사실은 주문이 스냅샷으로 보존하고, 장바구니는 현재 상품 상태를 보여준다.

## 5. 상품 정보 표시 정책

장바구니는 조회할 때 상품과 SKU의 현재 정보를 다시 읽어 보여준다.

- 담은 뒤 상품명이 변경되면 변경된 이름을 보여준다.
- 담은 뒤 가격이 변경되면 현재 판매가를 보여준다.
- 담은 뒤 재고가 부족해지면 품절 상태로 보여준다.
- 담은 뒤 상품이 판매중지되거나 단종되면 구매 불가 상태로 보여준다.
- 담은 뒤 브랜드가 비활성화되면 구매 불가 상태로 보여준다.
- 구매 불가 라인은 장바구니에서 자동 삭제하지 않고 표시한다.

자동 삭제하지 않는 이유는 고객이 왜 결제할 수 없는지 확인하고 직접 정리할 수 있어야 하기 때문이다.

## 6. 담기 정책

고객은 구매 가능한 SKU만 장바구니에 담을 수 있다.

담기 시점에는 다음 조건을 확인한다.

- SKU가 존재해야 한다.
- SKU가 속한 상품이 판매중이어야 한다.
- SKU가 속한 상품의 브랜드가 활성 상태여야 한다.
- 요청 수량은 1개 이상이어야 한다.
- 현재 판매 가능 재고가 요청 수량 이상이어야 한다.

조건을 만족하지 않으면 장바구니에 담을 수 없다.
담기 시점 검증은 장바구니가 명백히 구매 불가능한 항목으로 시작하지 않도록 하기 위한 조언적 검증이다.
최종 재고 차감과 구매 가능 여부 확정은 주문 생성 시점에 다시 수행한다.

## 7. 수량 정책

장바구니 라인의 수량은 1개 이상이어야 한다.

- 같은 SKU를 다시 담으면 기존 라인의 수량에 합산한다.
- 수량 변경은 새 수량으로 덮어쓰는 절대값 변경이다.
- 수량을 0으로 변경할 수 없다.
- 라인을 제거하려면 수량을 0으로 바꾸는 대신 제거 기능을 사용한다.
- 수량 변경 후 재고가 부족하면 변경할 수 없다.
- 한 장바구니에 담을 수 있는 서로 다른 SKU 라인 수에는 상한을 둔다.

같은 SKU를 합산하는 이유는 SKU가 이미 옵션 조합까지 확정된 판매 단위이기 때문이다.
같은 SKU를 여러 줄로 나누면 고객과 시스템 모두 수량 의미를 해석하기 어려워진다.

## 8. 조회 정책

장바구니 조회는 현재 구매 가능 상태를 함께 보여준다.

라인 상태는 다음과 같이 나뉜다.

| 상태 | 의미 |
|---|---|
| 구매 가능 | 상품이 판매중이고 브랜드가 활성 상태이며 SKU 재고가 요청 수량 이상이다 |
| 품절 | 상품이 판매중이고 브랜드가 활성 상태지만 SKU 재고가 요청 수량보다 부족하다 |
| 구매 불가 | 상품이 판매중이 아니거나 브랜드가 비활성이거나 상품/SKU 정보를 찾을 수 없다 |

장바구니 총액은 구매 가능한 라인의 현재 판매가 소계만 합산한다.
품절 또는 구매 불가 라인은 총액에 포함하지 않는다.

## 9. 주문 연결 정책

장바구니는 주문을 직접 생성하지 않는다.

현재 정책에서는 클라이언트가 장바구니를 조회한 뒤, 선택한 라인을 기존 주문 API로 전달한다.
주문 성공 후 장바구니 라인 제거 또는 비우기도 클라이언트가 호출한다.

이렇게 분리하는 이유는 다음과 같다.

- 주문 도메인이 장바구니를 알지 않아도 된다.
- 장바구니 검증과 주문 검증의 책임이 분리된다.
- 최종 재고 차감과 결제 처리는 주문 도메인이 권위 있게 수행한다.

인증과 체크아웃 정책이 강화되면 단일 체크아웃 API를 별도로 검토한다.

## 10. 삭제와 정리 정책

고객은 장바구니 라인을 제거하거나 장바구니를 비울 수 있다.

- 특정 SKU 라인을 제거할 수 있다.
- 장바구니 전체를 비울 수 있다.
- 장바구니를 비워도 회원이나 상품에는 영향을 주지 않는다.
- 버려진 장바구니를 자동 정리하는 배치는 현재 보류한다.

장바구니는 데이터베이스에 저장되므로 장기간 방치된 카트가 누적될 수 있다.
운영 단계에서는 마지막 수정 시각을 기준으로 정리 배치를 추가해야 한다.

## 11. 예외와 보류 정책

현재 정책에서 의식적으로 보류한 항목은 다음과 같다.

| 항목 | 현재 결정 | 향후 필요성 |
|---|---|---|
| 게스트 장바구니 | 보류 | 비회원 구매나 로그인 전 담기가 필요하면 쿠키·세션 기반 식별 정책이 필요하다 |
| 인증·인가 | 보류 | 회원 본인의 장바구니만 조회·변경할 수 있어야 한다 |
| 버려진 카트 정리 | 보류 | DB 저장 방식이므로 오래된 카트를 정리하는 배치가 필요하다 |
| 가격 변동 알림 | 보류 | 담은 시점 가격을 저장해야 하므로 현재 live 참조 원칙과 별도 정책이 필요하다 |
| 단일 체크아웃 API | 보류 | 주문 성공 후 카트 정리를 서버가 보장해야 할 때 추가한다 |
| 부분 선택 체크아웃 | 보류 | 장바구니 일부 라인만 주문하는 선택 정책이 필요하다 |
| 위시리스트 | 범위 밖 | 구매 후보 보관과 장기 저장 목록은 별도 정책으로 다룬다 |
