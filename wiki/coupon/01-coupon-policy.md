# 쿠폰 정책

작성: 2026-06-14 · v2 증분: 2026-06-18 · 상태: v1 구현 완료, v2 구현 완료

이 문서는 쿠폰이 고객에게 어떤 혜택으로 제공되고, 어떤 기준으로 발급·사용·복원되는지를 정의한다.
구현 구조와 기술적 근거는 [`coupon-domain-spec.md`](./coupon-domain-spec.md)를 따른다.

> **v2 증분(2026-06-18).** 두 가지 쿠폰 성격을 더한다. **적용 범위 한정 쿠폰**: 할인이 특정 브랜드·상품·카테고리 항목에만 붙는다(§4). **등급별 차등 할인**: 같은 쿠폰이라도 회원 등급에 따라 할인이 다르며, 그 차등은 발급받는 순간의 등급으로 고정된다(§4-A). 카테고리 한정은 계층형 카테고리 도메인 도입 후 추가했고, 상위 카테고리를 고르면 그 하위 카테고리 상품까지 덮는다(서브트리, §4).

## 1. 목적

쿠폰 정책은 주문 할인 혜택의 발급, 보유, 사용, 만료, 취소 복원 규칙을 정의한다.

이 정책은 다음 질문에 답한다.

- 쿠폰 정책과 발급 쿠폰은 각각 무엇을 의미하는가?
- 고객은 어떤 조건에서 쿠폰을 발급받을 수 있는가?
- 쿠폰 할인은 주문의 어느 항목에 적용되는가(주문 전체 / 브랜드 / 상품 / 카테고리)?
- 주문에서 쿠폰은 어떤 기준으로 할인액을 계산하는가?
- 같은 쿠폰이 회원 등급에 따라 어떻게 달라지며, 그 차등은 언제 확정되는가?
- 쿠폰은 언제 사용 처리되고, 언제 복원되는가?
- 쿠폰 할인 이후 주문 총액, 할인액, 청구액과 환불 금액은 무엇을 기준으로 하는가?

## 2. 핵심 용어

| 용어 | 의미 |
|---|---|
| 쿠폰 정책 | 쿠폰 이름, 할인 규칙, 적용 범위, 발급 가능 기간, 발급 수량, 유효일수 등을 정의한 캠페인 |
| 발급 쿠폰 | 특정 회원에게 발급된 쿠폰 인스턴스 |
| 할인 규칙 | 정액 또는 정률 할인, 최소 주문 금액, 최대 할인 한도를 포함한 계산 규칙 |
| 적용 범위 | 할인이 어느 주문 항목에 붙는지의 기준. 주문 전체 / 특정 브랜드 / 특정 상품 / 특정 카테고리 |
| 한정 쿠폰 | 적용 범위가 주문 전체가 아니라 특정 브랜드·상품·카테고리로 좁혀진 쿠폰 |
| 카테고리 서브트리 | 어떤 카테고리와 그 아래 모든 하위 카테고리를 묶은 범위. 카테고리 한정 쿠폰은 대상 카테고리의 서브트리에 속한 상품을 매칭한다 |
| 할인 대상 금액 | 할인이 실제로 계산되는 기준 금액. 주문 전체 쿠폰은 주문 총액, 한정 쿠폰은 매칭된 항목들의 부분합 |
| 정액 할인 | 할인 대상 금액에서 고정 금액을 차감하는 할인 |
| 정률 할인 | 할인 대상 금액의 일정 비율을 차감하는 할인 |
| 최소 주문 금액 | 쿠폰을 사용할 수 있는 할인 대상 금액 기준 |
| 최대 할인 한도 | 계산된 할인액이 넘을 수 없는 선택 상한. 주로 정률 할인에 쓰지만, 현재 구현은 정액 할인에도 값이 있으면 같은 cap으로 적용한다 |
| 회원 등급 | 회원의 등급 구분. BRONZE / SILVER / GOLD / VIP 네 단계 |
| 등급별 차등 할인 | 같은 쿠폰 정책이라도 회원 등급에 따라 할인 규칙이 달라지는 것 |
| 사용 | 주문 생성 과정에서 쿠폰을 소비해 중복 사용을 막는 처리 |
| 복원 | 결제 실패나 주문 취소로 사용 처리된 쿠폰을 다시 미사용 상태로 돌리는 처리 |
| 만료 | 쿠폰 유효 기간이 지나 더 이상 사용할 수 없는 상태 |

## 3. 쿠폰 정책과 발급 쿠폰

쿠폰 정책은 운영자가 만드는 캠페인 단위이고, 발급 쿠폰은 회원이 실제로 보유하는 사용 단위다.

- 쿠폰 정책은 여러 회원에게 공유되는 발급 기준이다.
- 발급 쿠폰은 한 회원에게 속한 개별 쿠폰이다.
- 고객이 주문에 사용하는 것은 쿠폰 정책이 아니라 발급 쿠폰이다.
- 발급 쿠폰은 발급 시점의 할인 규칙, 적용 범위, 만료 시각을 보존한다.
- 등급별 차등 할인은 발급 시점의 회원 등급으로 규칙을 확정해 보존한다(§4-A).
- 쿠폰 정책이 나중에 변경되거나 회원 등급이 바뀌어도 이미 발급된 쿠폰의 할인 규칙과 적용 범위는 바뀌지 않는다.

이렇게 분리하는 이유는 운영자가 관리하는 캠페인과 고객이 보유한 혜택의 생명주기가 다르기 때문이다.
정책은 발급 수량과 기간을 관리하고, 발급 쿠폰은 소유자와 사용 상태를 관리한다.

## 4. 할인 정책

쿠폰마다 **적용 범위**가 있고, 할인은 그 범위에 해당하는 금액에만 적용한다.

적용 범위는 넷 중 하나다.

| 적용 범위 | 할인 대상 | 예 |
|---|---|---|
| 주문 전체 | 주문 항목 전체의 합 | "전 상품 10% 할인" |
| 브랜드 한정 | 특정 브랜드 항목들의 부분합 | "나이키 상품 20% 할인" |
| 상품 한정 | 특정 상품 항목들의 부분합 | "이 운동화 5천원 할인" |
| 카테고리 한정 | 특정 카테고리(서브트리) 항목들의 부분합 | "신발 카테고리 15% 할인" |

- 한 쿠폰은 적용 범위를 하나만 가진다. 한 쿠폰이 여러 브랜드나 "브랜드이면서 카테고리" 같은 교차 조건을 동시에 겨냥하지 않는다.
- 한 주문에는 쿠폰을 최대 1장만 사용할 수 있다.
- 한정 쿠폰의 할인은 **매칭된 항목들의 부분합**을 기준으로 계산한다. 매칭되지 않는 다른 항목은 할인 대상이 아니다.
- 주문에 쿠폰의 적용 범위에 맞는 항목이 하나도 없으면 그 쿠폰은 사용할 수 없다. 조용히 0원 할인으로 통과시키지 않고 사용을 거부한다.
- 쿠폰 할인은 정액 할인과 정률 할인을 지원한다.
- 정률 할인율은 정수 퍼센트 단위다.
- 정률 할인 계산에서 소수점 이하는 버린다.
- 할인 규칙은 최대 할인 한도를 가질 수 있다. 주 사용처는 정률 할인이지만, 현재 구현은 정액 할인에도 값이 있으면 계산된 할인액을 그 한도로 줄인다.
- 최대 할인 한도가 없으면 계산된 할인액을 그대로 사용한다.
- 할인액이 할인 대상 금액보다 크면 할인 대상 금액까지만 할인한다.
- 쿠폰 할인으로 청구 금액은 0원이 될 수 있지만 음수가 될 수는 없다.

브랜드 한정은 "그 브랜드 상품을 산 만큼만 깎아준다"는 운영 의도를 그대로 표현한다.
주문에 그 브랜드가 있다고 주문 전체를 깎으면 다른 브랜드까지 혜택이 새기 때문에, 할인은 매칭 항목 부분합에만 건다.

카테고리 한정은 대상 카테고리와 그 아래 모든 하위 카테고리(서브트리)에 속한 상품을 매칭한다.

- 상품은 최하위 카테고리에만 등록되므로, 상위 카테고리를 고른 쿠폰은 그 아래 하위 카테고리 상품들을 덮는다. 예를 들어 "신발" 쿠폰은 "신발 > 운동화 > 러닝화"와 "신발 > 구두 > 더비"에 등록된 상품을 모두 포함한다.
- 최하위 카테고리를 직접 고르면 그 카테고리 상품만 매칭한다.
- 쿠폰이 덮는 범위는 **주문 시점의 현재 카테고리 구조**를 기준으로 판단한다. 쿠폰을 발급받은 뒤 그 카테고리 아래에 새 하위 카테고리나 상품이 추가되면, 그 상품도 매칭 대상이 된다.

서브트리로 매칭하는 이유는 상품이 최하위 카테고리에만 매여 있어, 상위 카테고리를 정확 일치로만 보면 어떤 상품도 걸리지 않기 때문이다.
"신발 카테고리 할인"은 신발 아래 모든 종류를 덮는다는 운영 의도를 서브트리가 그대로 표현한다.
발급 시점이 아니라 주문 시점의 카테고리 구조로 판단하는 이유는, 카테고리에 무엇이 속하는지는 고정된 약속이 아니라 그때그때의 구조적 사실이기 때문이다. 할인율 같은 혜택은 발급 시점으로 고정하지만(§4-A), "그 카테고리에 무엇이 들어 있나"는 쓰는 시점의 최신 상태를 따른다.

## 4-A. 등급별 차등 할인 정책

같은 쿠폰 정책이라도 회원 등급에 따라 할인 규칙이 다를 수 있다.

- 쿠폰 정책은 **기본 할인 규칙** 하나와, 일부 등급에 대한 **등급별 할인 규칙**을 가질 수 있다.
- 회원이 쿠폰을 발급받을 때, 그 회원 등급에 해당하는 등급별 규칙이 있으면 그 규칙을, 없으면 기본 규칙을 적용한다.
- 등급별 규칙은 모든 등급을 채울 필요가 없다. 비워 두면 그 등급은 기본 규칙을 받는다.
- 등급 차등이 없는 일반 쿠폰은 등급별 규칙이 전부 비어 있는 경우와 같다.
- 등급 차등은 **발급받는 순간의 등급으로 확정**된다. 발급 이후 회원 등급이 바뀌어도 이미 발급된 쿠폰의 혜택은 그대로다.
- 적용 범위와 등급 차등은 서로 독립이다. "GOLD가 받으면 20%, 그 외 10% — 단 나이키 상품에만" 같은 조합이 가능하다.
- 등급에 따라 할인 규칙(할인율·금액·한도)만 달라지고, 적용 범위는 등급과 무관하게 정책이 정한 그대로다.

혜택을 발급 시점 등급으로 고정하는 이유는, 고객이 받은 쿠폰의 혜택이 보유 중에 흔들리지 않게 하기 위해서다.
"내가 받을 때 20% 쿠폰이었으면 쓸 때도 20%"가 쿠폰이라는 약속에 자연스럽다.

회원 등급은 운영자 수동 지정 전제의 정적 값이며, 누적 구매액 등으로 자동 산정하지 않는다(§12). 현재 코드에는 등급 변경 도메인 메서드는 있지만 외부 운영 API는 없다.

## 5. 최소 주문 금액 정책

쿠폰은 할인 대상 금액이 최소 주문 금액 이상일 때만 사용할 수 있다.

- 최소 주문 금액은 **할인 대상 금액**을 기준으로 판단한다. 주문 전체 쿠폰은 주문 총액, 한정 쿠폰은 매칭 항목 부분합이 기준이다.
- 한정 쿠폰은 자격 판단과 할인 계산이 같은 금액(매칭 부분합)을 본다. 예를 들어 "나이키 5만원 이상 시 할인"은 나이키 상품을 5만원어치 이상 담아야 적용된다. 주문 전체가 5만원을 넘어도 나이키가 그에 못 미치면 사용할 수 없다.
- 할인 후 청구 금액은 최소 주문 금액 판단에 사용하지 않는다.
- 최소 주문 금액이 0원이면 매칭 항목이 있는 한 금액과 무관하게 사용할 수 있다.
- 할인 대상 금액이 최소 주문 금액보다 작으면 쿠폰을 사용할 수 없다.

자격과 혜택을 같은 금액으로 판단하는 이유는, 한정 쿠폰의 "이 범위를 이만큼 사면 이만큼 깎아준다"는 조건을 일관되게 지키기 위해서다.

## 6. 발급 정책

현재 쿠폰 발급 방식은 회원 선착순 다운로드다.

- 회원은 발급 가능한 쿠폰 정책에 대해 직접 쿠폰을 발급받을 수 있다.
- 쿠폰 정책은 발급 가능 시작 시각과 종료 시각을 가진다.
- 발급 가능 기간 밖에서는 쿠폰을 발급받을 수 없다.
- 쿠폰 정책이 비활성 상태이면 발급 기간 안이어도 쿠폰을 발급받을 수 없다.
- 쿠폰 정책별 최대 발급 수량을 초과해 발급할 수 없다.
- 한 회원은 같은 쿠폰 정책에서 쿠폰을 1장만 발급받을 수 있다.
- 이미 발급받은 회원이 다시 발급을 요청하면 중복 발급으로 거부한다.
- 발급 시 회원을 조회해 그 회원의 등급을 읽고, 등급별 차등 할인 규칙을 확정한다(§4-A). 회원 조회 과정에서 회원 존재가 검증되므로, 존재하지 않는 회원으로는 발급할 수 없다.

선착순 발급은 "먼저 요청한 회원에게 제한된 수량의 혜택을 제공한다"는 운영 의도를 갖는다.
발급 수량 초과와 회원 중복 발급은 동시에 많은 요청이 들어와도 허용하지 않는다.
발급 시점에 회원 등급을 읽어야 하므로 발급은 회원 존재를 검증한다.
쿠폰 소유자는 요청 body의 `memberId`가 아니라 인증된 회원 ID로 결정한다.

## 7. 유효 기간과 만료 정책

쿠폰의 유효 기간은 발급일을 기준으로 계산한다.

- 발급 쿠폰은 발급 시각을 기록한다.
- 만료 시각은 발급일에 유효일수를 더한 날짜의 자정이다.
- 쿠폰은 만료 시각보다 이전인 동안만 사용할 수 있다.
- 만료 시각과 같은 순간부터는 사용할 수 없다.
- 만료는 별도 저장 상태가 아니라 현재 시각과 만료 시각 비교로 판단한다.
- 만료된 쿠폰을 별도 배치로 사용 불가 상태로 전환하지 않는다.

예를 들어 6월 14일에 유효일수 7일 쿠폰을 발급받으면, 6월 21일 00:00부터 사용할 수 없다.
시·분 단위의 발급 시각에 따라 고객별 유효 기간 체감이 달라지지 않도록 날짜 단위 만료를 사용한다.

## 8. 사용 정책

쿠폰은 주문 생성 과정에서 사용 처리된다.

- 주문 요청에 쿠폰 ID가 없으면 쿠폰 없이 주문한다.
- 주문 요청에 쿠폰 ID가 있으면 해당 쿠폰이 존재해야 한다.
- 쿠폰 소유자는 주문자와 같아야 한다.
- 소유자 검증은 인증된 주문자 ID와 발급 쿠폰의 `memberId`가 같은지 비교하는 방식이다.
- 미사용 쿠폰만 사용할 수 있다.
- 만료되지 않은 쿠폰만 사용할 수 있다.
- 쿠폰의 적용 범위에 맞는 항목이 주문에 하나 이상 있어야 한다. 한정 쿠폰인데 해당 브랜드·상품·카테고리(서브트리) 항목이 주문에 없으면 사용할 수 없다.
- 할인 대상 금액(주문 전체 쿠폰은 주문 총액, 한정 쿠폰은 매칭 부분합)이 쿠폰의 최소 주문 금액을 만족해야 한다.
- 쿠폰 사용이 성공하면 쿠폰은 사용 상태가 되고, 사용한 주문 ID를 기록한다.
- 이미 사용된 쿠폰은 다른 주문에 다시 사용할 수 없다.

쿠폰을 결제 성공 이후가 아니라 주문 생성 과정에서 사용 처리하는 이유는 결제 대기 중 같은 쿠폰이 다른 주문에 중복 사용되는 상황을 막기 위해서다.

## 9. 주문 금액 정책

쿠폰이 적용된 주문은 주문 총액, 할인액, 청구액을 모두 보존한다.

| 금액 | 의미 |
|---|---|
| 주문 총액 | 주문 항목별 주문 시점 판매가와 수량을 곱해 합산한 할인 전 금액 |
| 할인액 | 쿠폰으로 차감된 금액 |
| 청구액 | 주문 총액에서 할인액을 뺀 실제 결제·환불 기준 금액 |

- 쿠폰이 없으면 할인액은 0원이고 청구액은 주문 총액과 같다.
- 쿠폰이 있으면 할인액과 청구액을 주문에 함께 기록한다.
- 결제 요청은 주문 총액이 아니라 청구액을 기준으로 한다.
- 환불 요청도 주문 총액이 아니라 청구액을 기준으로 한다.
- 청구액이 0원이면 결제 게이트웨이 호출을 생략하고 결제 완료로 처리한다.

세 금액을 모두 보존하는 이유는 고객에게 할인 전 금액, 할인 혜택, 실제 청구액을 모두 설명할 수 있어야 하기 때문이다.

## 10. 취소와 복원 정책

결제 실패 또는 주문 취소가 발생하면 사용했던 쿠폰을 복원한다.

- 쿠폰 적용 주문의 결제가 실패하면 주문을 취소하고 쿠폰을 미사용 상태로 복원한다.
- 쿠폰 적용 주문을 고객이 취소하면 쿠폰을 미사용 상태로 복원한다.
- 쿠폰 복원 시 사용 주문 ID를 제거한다.
- 이미 복원된 쿠폰을 다시 복원하려는 요청은 결과적으로 아무 변화가 없어야 한다.
- 만료된 쿠폰이 복원되어도 다시 사용할 수 있는 것은 아니다. 만료 여부는 사용 시점에 다시 판단한다.

쿠폰을 복원하는 이유는 주문이 무효화되었을 때 고객이 실제로 혜택을 소비한 것으로 보지 않기 때문이다.
환불과 재고 복원처럼 쿠폰도 주문 실패·취소의 보상 대상이다.

## 11. 조회 정책

회원은 자신에게 발급된 쿠폰 목록을 조회할 수 있다.

- 쿠폰 목록은 회원 ID를 기준으로 조회한다.
- 회원 쿠폰 목록 API는 `/api/v1/coupons/me`이며 인증된 회원 ID 기준으로 조회한다.
- 쿠폰 상태로 목록을 필터링할 수 있다.
- 응답에는 쿠폰 상태와 만료 여부를 함께 보여줄 수 있다.
- 만료 여부는 조회 시점에 계산한다.
- 조회 요청의 회원 ID가 실제 회원 테이블에 존재하는지 별도로 재검증하지는 않는다.

고객 화면에서 "사용 가능", "사용 완료", "기간 만료"처럼 보이는 값은 저장 상태와 만료 여부를 조합해 표현할 수 있다.

## 12. 예외와 보류 정책

현재 정책에서 의식적으로 보류한 항목은 다음과 같다.

| 항목 | 현재 결정 | 향후 필요성 |
|---|---|---|
| 상품/브랜드 한정 쿠폰 | **v2 적용** | 상품·브랜드 한정을 적용 범위(scope)로 구현(§4) |
| 등급별 차등 할인 | **v2 적용** | 발급 시점 등급으로 할인 규칙을 확정(§4-A) |
| 카테고리 한정 쿠폰 | **v2 적용** | 계층형 카테고리 도메인 도입 후 서브트리 매칭으로 구현(§4). 주문 시점의 현재 카테고리 구조로 판단 |
| 다중·교차 대상 쿠폰 | 보류 | 여러 브랜드/상품이나 "브랜드이면서 카테고리" 같은 교차 조건을 한 쿠폰에 담으려면 규칙 조합 정책이 필요하다 |
| 등급 발급 자격 제한 | 보류 | "특정 등급만 발급받을 수 있는" 게이트는 등급별 차등 할인과는 별개 축이다. 필요하면 발급 자격 조건을 따로 둔다 |
| 등급별 적용 범위 차등 | 보류 | 등급에 따라 적용 범위까지 달라지는 쿠폰. 적용 범위는 캠페인의 정체성이라 등급과 무관하게 고정한다 |
| 회원 등급 자동 산정 | 보류 | 누적 구매액·주문 이력 기반 승급/강등. 현재 등급은 도메인에서 수동 지정 전제이며, 외부 등급 변경 API는 아직 없다 |
| 쿠폰 중복 적용 | 보류 | 여러 쿠폰의 적용 순서, 최대 할인, 중복 제한 정책이 필요하다 |
| 부분 취소 | 보류 | 라인별 할인 배분과 부분 환불 시 쿠폰 복원 여부를 정해야 한다 |
| 프로모 코드 | 보류 | 코드 입력, 노출 범위, 코드 유출 대응 정책이 필요하다 |
| 관리자 직접 발급 | 보류 | CS 보상, 타겟 발급, 대량 발급 정책이 필요하다 |
| 고정 만료 기간 | 보류 | 이벤트 기간 고정형 쿠폰이 필요하면 상대 유효기간과 병행 정책이 필요하다 |
| 쿠폰 사용 멱등성 | 보류 | 주문 재시도 시 같은 쿠폰을 안전하게 재사용/재확인하는 정책이 필요하다 |
| 고트래픽 선착순 | 보류 | Redis 등 빠른 카운터와 DB 정합성 보정 정책이 필요하다 |
| 인증·인가 | 적용 | 회원 본인의 쿠폰만 인증 회원 ID 기준으로 발급·조회·사용한다 |
| 회원 존재 검증 | 발급은 검증, 조회는 보류 | 발급 시 회원을 조회해 등급을 읽으므로 회원 존재가 검증된다(§6). 쿠폰 목록 조회는 아직 회원 존재를 검증하지 않는다 |
| 운영자 권한 | 적용 | 쿠폰 정책 생성은 `ADMIN` 권한으로 제한한다 |

## 13. 구현 참고

이 문서는 비즈니스 정책을 설명한다.
구현 방식, Aggregate 구조, 트랜잭션 경계, 동시성 전략, 계층 매핑은 [`coupon-domain-spec.md`](./coupon-domain-spec.md)를 기준으로 한다.

주문 금액과 결제·취소 흐름은 주문 정책과 함께 읽어야 한다.
쿠폰 도입 이후 주문의 결제·환불 기준 금액은 청구액이며, 관련 주문 정책은 [`../order/01-order-policy.md`](../order/01-order-policy.md)를 따른다.
