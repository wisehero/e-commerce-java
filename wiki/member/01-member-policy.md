# 회원 정책

작성: 2026-06-13 · 상태: 구현 완료(이번 이터레이션 범위)

이 문서는 회원이 시스템에 가입하고 식별되는 기준, 인증 정보와 권한의 기본 의미를 정의한다.
현재 코드는 회원가입 정책을 중심으로 구현되어 있으며, 로그인·탈퇴·권한 변경은 아직 다루지 않는다.

## 1. 목적

회원 정책은 고객이 서비스의 회원으로 등록되기 위해 충족해야 하는 조건과, 가입 후 시스템이 회원을 어떻게 식별하고 보호하는지를 정의한다.

이 정책은 다음 질문에 답한다.

- 어떤 이메일을 회원 식별자로 인정하는가?
- 비밀번호는 어떤 기준으로 받고 저장하는가?
- 닉네임은 어떤 조건을 만족해야 하는가?
- 신규 회원은 어떤 권한으로 시작하는가?
- 중복 회원가입은 어떻게 막는가?

## 2. 핵심 용어

| 용어 | 의미 |
|---|---|
| 회원 | 이메일, 비밀번호, 닉네임, 권한을 가진 서비스 사용자 |
| 이메일 | 회원을 식별하는 로그인용 주소 |
| 비밀번호 | 회원 본인 확인에 사용하는 비밀 정보 |
| 비밀번호 해시 | 평문 비밀번호를 직접 저장하지 않기 위해 변환한 값 |
| 닉네임 | 서비스 안에서 회원을 표시하는 이름 |
| 권한 | 회원이 수행할 수 있는 행위의 범위 |
| 일반 회원 | 기본 고객 권한을 가진 회원 |
| 관리자 | 운영 행위를 수행할 수 있는 권한을 가진 회원 |

## 3. 회원가입 정책

회원가입은 이메일, 비밀번호, 닉네임을 받아 새 회원을 등록하는 행위다.

- 이메일은 필수다.
- 비밀번호는 필수다.
- 닉네임은 필수다.
- 이미 가입된 이메일로는 다시 가입할 수 없다.
- 이미 사용 중인 닉네임으로는 가입할 수 없다.
- 신규 회원은 일반 회원 권한으로 가입된다.

중복 검사는 이메일을 먼저 확인한 뒤 닉네임을 확인한다.
이메일이 이미 가입되어 있으면 닉네임 중복 여부는 확인하지 않고 가입을 거부한다.

## 4. 이메일 정책

이메일은 회원을 식별하는 주요 값이다.

- 이메일은 비어 있을 수 없다.
- 이메일은 유효한 이메일 형식이어야 한다.
- 이메일은 회원마다 유일해야 한다.
- 같은 이메일로 여러 회원을 만들 수 없다.

현재 이메일 형식은 일반적인 `local@domain.tld` 형태를 기준으로 검증한다.

## 5. 비밀번호 정책

비밀번호는 회원 본인 확인에 사용하는 비밀 정보다.

- 평문 비밀번호는 8자 이상이어야 한다.
- 평문 비밀번호는 저장하지 않는다.
- 저장 시 비밀번호 해시값으로 변환한다.
- 비밀번호 해시값은 비어 있을 수 없다.
- 회원 응답에는 비밀번호나 비밀번호 해시값을 포함하지 않는다.

현재 비밀번호 해시는 BCrypt를 사용한다.
비밀번호를 문자열로 표현해야 하는 경우에도 실제 해시값이 노출되지 않도록 감춘다.

## 6. 닉네임 정책

닉네임은 서비스 안에서 회원을 표시하는 이름이다.

- 닉네임은 2자 이상 20자 이하여야 한다.
- 닉네임은 회원마다 유일해야 한다.
- 같은 닉네임으로 여러 회원을 만들 수 없다.
- 닉네임 변경 시에도 동일한 길이 규칙을 적용한다.

현재 회원가입 흐름에서는 닉네임 중복을 검사한다.
닉네임 변경 기능은 도메인 규칙만 존재하며, 별도 유스케이스와 API는 아직 없다.

## 7. 권한 정책

회원은 권한을 가진다.

| 권한 | 의미 |
|---|---|
| 일반 회원 | 기본 가입 회원. 고객 행위를 수행한다. |
| 관리자 | 운영 행위를 수행할 수 있는 회원이다. |

- 신규 가입 회원은 일반 회원 권한으로 시작한다.
- 권한은 필수다.
- 관리자인지 여부는 회원의 권한으로 판단한다.

현재 가입 요청으로 관리자 회원을 만들 수 없다.
관리자 부여, 권한 변경, 권한 회수 정책은 아직 다루지 않는다.

## 8. 응답 정책

회원가입이 성공하면 가입된 회원 정보를 반환한다.

응답에는 다음 정보를 포함한다.

- 회원 ID
- 이메일
- 닉네임
- 권한

응답에는 다음 정보를 포함하지 않는다.

- 평문 비밀번호
- 비밀번호 해시값

## 9. 예외와 보류 정책

현재 정책에서 의식적으로 보류한 항목은 다음과 같다.

| 항목 | 현재 결정 | 향후 필요성 |
|---|---|---|
| 로그인 | 보류 | 이메일과 비밀번호로 세션 또는 토큰을 발급해야 한다 |
| 로그아웃 | 보류 | 발급된 인증 상태를 종료하는 정책이 필요하다 |
| 회원 탈퇴 | 보류 | 주문 이력 보존과 개인정보 삭제 범위를 정해야 한다 |
| 비밀번호 변경 | 보류 | 기존 비밀번호 확인과 새 비밀번호 검증 정책이 필요하다 |
| 비밀번호 재설정 | 보류 | 이메일 인증, 재설정 토큰, 만료 정책이 필요하다 |
| 이메일 변경 | 보류 | 새 이메일 중복 확인과 본인 확인 정책이 필요하다 |
| 닉네임 변경 API | 보류 | 현재 도메인 규칙만 있고 외부 기능은 없다 |
| 관리자 권한 부여 | 보류 | 누가 어떤 절차로 관리자 권한을 부여할지 정해야 한다 |
| 계정 상태 | 보류 | 휴면, 정지, 탈퇴 같은 상태 정책이 필요할 수 있다 |

## 10. 구현 참고

이 문서는 비즈니스 정책을 설명한다.
현재 구현은 회원가입 API와 회원 도메인 규칙을 중심으로 한다.

비밀번호 해시는 `PasswordHasher` 포트를 통해 처리하며, 현재 구현은 BCrypt를 사용한다.
