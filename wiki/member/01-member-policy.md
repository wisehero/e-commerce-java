# 회원 정책

작성: 2026-06-13 · 상태: 회원가입 구현 완료, 인증·인가 결정 완료, 인증 경계 일부 구현

이 문서는 회원이 시스템에 가입하고 식별되는 기준, 인증 정보와 권한·등급의 기본 의미를 정의한다.
현재 코드는 회원가입, 회원 등급, 회원 상태·로그인 실패 횟수·인증 버전 도메인 규칙, 공개/보호/관리자 API 보안 경계, 인증 principal 기반 회원 ID 주입을 구현한다.
로그인·refresh·logout·me API, 실제 access JWT 발급·검증 필터, refresh token 저장소, rate limit, 감사 로그, 탈퇴·권한 변경·등급 변경 API는 아직 후속 구현 대상이다.
인증·인가 설계 결정은 [`02-authentication-authorization-adr.md`](./02-authentication-authorization-adr.md)를 따른다.
구현 순서와 검증 계획은 [`03-authentication-implementation-plan.md`](./03-authentication-implementation-plan.md)를 따른다.

## 1. 목적

회원 정책은 고객이 서비스의 회원으로 등록되기 위해 충족해야 하는 조건과, 가입 후 시스템이 회원을 어떻게 식별하고 보호하는지를 정의한다.

이 정책은 다음 질문에 답한다.

- 어떤 이메일을 회원 식별자로 인정하는가?
- 비밀번호는 어떤 기준으로 받고 저장하는가?
- 닉네임은 어떤 조건을 만족해야 하는가?
- 신규 회원은 어떤 권한으로 시작하는가?
- 신규 회원은 어떤 등급으로 시작하는가?
- 중복 회원가입은 어떻게 막는가?
- 로그인은 어떤 방식으로 처리하는가?
- 로그인 실패 누적과 회원 상태는 어떻게 관리하는가?

## 2. 핵심 용어

| 용어 | 의미 |
|---|---|
| 회원 | 이메일, 비밀번호, 닉네임, 권한, 등급을 가진 서비스 사용자 |
| 이메일 | 회원을 식별하는 로그인용 주소 |
| 비밀번호 | 회원 본인 확인에 사용하는 비밀 정보 |
| 비밀번호 해시 | 평문 비밀번호를 직접 저장하지 않기 위해 변환한 값 |
| 닉네임 | 서비스 안에서 회원을 표시하는 이름 |
| 권한 | 회원이 수행할 수 있는 행위의 범위 |
| 회원 등급 | 쿠폰 등급별 차등 할인에서 발급 시점 기준으로 읽는 정적 등급 |
| 일반 회원 | 기본 고객 권한을 가진 회원 |
| 관리자 | 운영 행위를 수행할 수 있는 권한을 가진 회원 |
| 회원 상태 | 로그인 가능 여부와 운영 제재·탈퇴 여부를 나타내는 상태 |
| 자동로그인 | refresh token 만료 기간을 길게 두어 로그인 상태를 오래 유지하는 기능 |
| 인증 버전 | 기존 access token을 무효화하기 위한 회원별 버전 값 |

## 3. 회원가입 정책

회원가입은 이메일, 비밀번호, 닉네임을 받아 새 회원을 등록하는 행위다.

- 이메일은 필수다.
- 비밀번호는 필수다.
- 닉네임은 필수다.
- 이미 가입된 이메일로는 다시 가입할 수 없다.
- 이미 사용 중인 닉네임으로는 가입할 수 없다.
- 신규 회원은 일반 회원 권한으로 가입된다.
- 신규 회원은 `BRONZE` 등급으로 가입된다.

중복 검사는 이메일을 먼저 확인한 뒤 닉네임을 확인한다.
이메일이 이미 가입되어 있으면 닉네임 중복 여부는 확인하지 않고 가입을 거부한다.

## 4. 이메일 정책

이메일은 회원을 식별하는 주요 값이다.

- 이메일은 비어 있을 수 없다.
- 이메일은 유효한 이메일 형식이어야 한다.
- 이메일은 회원마다 유일해야 한다.
- 같은 이메일로 여러 회원을 만들 수 없다.

현재 이메일 형식은 일반적인 `local@domain.tld` 형태를 기준으로 검증한다.

## 5. 비밀번호 정책

비밀번호는 회원 본인 확인에 사용하는 비밀 정보다.

- 평문 비밀번호는 8자 이상이어야 한다.
- 평문 비밀번호는 저장하지 않는다.
- 저장 시 비밀번호 해시값으로 변환한다.
- 비밀번호 해시값은 비어 있을 수 없다.
- 회원 응답에는 비밀번호나 비밀번호 해시값을 포함하지 않는다.

현재 비밀번호 해시는 BCrypt를 사용한다.
비밀번호를 문자열로 표현해야 하는 경우에도 실제 해시값이 노출되지 않도록 감춘다.
정책 문서에서 "암호화"라고 표현하더라도 구현 의미는 복호화 가능한 암호화가 아니라 단방향 비밀번호 해시다.

## 6. 닉네임 정책

닉네임은 서비스 안에서 회원을 표시하는 이름이다.

- 닉네임은 2자 이상 20자 이하여야 한다.
- 닉네임은 회원마다 유일해야 한다.
- 같은 닉네임으로 여러 회원을 만들 수 없다.
- 닉네임 변경 시에도 동일한 길이 규칙을 적용한다.

현재 회원가입 흐름에서는 닉네임 중복을 검사한다.
닉네임 변경 기능은 도메인 규칙만 존재하며, 별도 유스케이스와 API는 아직 없다.

## 7. 권한 정책

회원은 권한을 가진다.

| 권한 | 의미 |
|---|---|
| 일반 회원 | 기본 가입 회원. 고객 행위를 수행한다. |
| 관리자 | 운영 행위를 수행할 수 있는 회원이다. |

- 신규 가입 회원은 일반 회원 권한으로 시작한다.
- 권한은 필수다.
- 관리자인지 여부는 회원의 권한으로 판단한다.

현재 가입 요청으로 관리자 회원을 만들 수 없다.
관리자 부여, 권한 변경, 권한 회수 정책은 아직 다루지 않는다.

### 회원 등급 정책

회원은 쿠폰 등급별 차등 할인을 위해 등급을 가진다.

| 등급 | 의미 |
|---|---|
| `BRONZE` | 신규 가입 기본 등급 |
| `SILVER` | 수동 지정 가능한 회원 등급 |
| `GOLD` | 수동 지정 가능한 회원 등급 |
| `VIP` | 수동 지정 가능한 회원 등급 |

- 등급은 필수다.
- 현재 신규 가입 회원은 `BRONZE` 등급으로 시작한다.
- 현재 도메인에는 수동 등급 변경 메서드(`changeGrade`)가 있지만, 외부 등급 변경 API는 없다.
- 현재 등급 간 서열 비교나 자동 산정 규칙은 없다. 쿠폰 정책의 등급별 override는 등급 값을 키로 조회한다.

## 8. 로그인 정책

이 절은 채택된 정책이다. 현재 코드에는 로그인 API와 토큰 발급·refresh token rotation 구현이 아직 없다.

로그인은 이메일과 비밀번호로 수행한다.

- 이메일은 회원 식별자로 사용한다.
- 비밀번호는 평문으로 저장하지 않고 저장된 비밀번호 해시와 비교한다.
- 로그인 성공 시 access token과 refresh token을 발급한다.
- 자동로그인을 요청하면 refresh token 만료 기간을 더 길게 둔다.
- 존재하지 않는 이메일과 비밀번호 불일치 응답은 외부에 구분해 노출하지 않는다.
- access token은 짧게 유지하고, refresh token은 회전 방식으로 재발급한다.
- 회원 상태·권한·비밀번호 변경, 전체 로그아웃 시 인증 버전을 증가시켜 기존 access token을 무효화한다.

상세 인증 방식은 [`02-authentication-authorization-adr.md`](./02-authentication-authorization-adr.md)를 따른다.

## 9. 로그인 실패 잠금 정책

현재 도메인 모델에는 로그인 실패 기록, 5회 실패 시 `LOCKED` 전환, 성공 시 실패 횟수 초기화 규칙이 있다.
다만 로그인 UseCase가 아직 없으므로 실제 HTTP 로그인 실패와 이 규칙은 아직 연결되지 않았다.
IP·이메일 기준 rate limit과 감사 로그도 미구현이다.

회원별 로그인 실패 횟수를 관리한다.

- 비밀번호 불일치로 로그인에 실패하면 실패 횟수를 1 증가시킨다.
- 로그인 실패 횟수가 5회에 도달하면 회원을 로그인 불가능 상태로 전환한다.
- 로그인 불가능 상태의 회원은 올바른 비밀번호를 입력해도 로그인할 수 없다.
- 로그인 성공 시 실패 횟수는 0으로 초기화한다.
- 로그인 실패는 계정별 누적을 기본으로 하며, IP·이메일 기준 rate limit은 후속 로그인 구현에서 연결한다.
- 로그인 실패, 성공, 잠금, 잠금 해제 감사 로그도 후속 구현 대상이다.

초기 상태명은 `LOCKED`를 사용한다.
잠금 해제 방식은 관리자 해제 또는 비밀번호 재설정 성공 중 하나로 별도 결정한다.

## 10. 회원 상태 정책

회원은 상태를 가진다.

| 상태 | 의미 | 로그인 |
|---|---|---:|
| `ACTIVE` | 정상 회원 | 가능 |
| `LOCKED` | 로그인 실패 5회로 잠긴 회원 | 불가 |
| `SUSPENDED` | 운영 정책으로 정지된 회원 | 불가 |
| `WITHDRAWN` | 탈퇴한 회원 | 불가 |

상태 필드와 상태 변경 시 인증 버전 증가 규칙은 구현되어 있다.
`LOCKED`, `SUSPENDED`, `WITHDRAWN` 상태가 되면 기존 refresh token을 폐기하고 기존 access token을 거부해야 한다는 정책은 채택됐지만, refresh token 저장소와 access token 검증 흐름은 아직 구현되어 있지 않다.
탈퇴 회원의 주문 이력 보존, 개인정보 마스킹, 동일 이메일 재가입 허용 여부는 별도 탈퇴 정책에서 정한다.

## 11. 응답 정책

회원가입이 성공하면 가입된 회원 정보를 반환한다.

응답에는 다음 정보를 포함한다.

- 회원 ID
- 이메일
- 닉네임
- 권한

응답에는 다음 정보를 포함하지 않는다.

- 평문 비밀번호
- 비밀번호 해시값
- 회원 등급
- 회원 상태
- 로그인 실패 횟수
- 인증 버전

회원 등급은 도메인과 JPA에는 저장되지만, 현재 회원가입 응답(`SignUpResponse`)에는 포함하지 않는다.

## 12. 예외와 보류 정책

현재 정책에서 의식적으로 보류한 항목은 다음과 같다.

| 항목 | 현재 결정 | 향후 필요성 |
|---|---|---|
| 로그인 | 결정 완료·구현 예정 | 이메일과 비밀번호로 access/refresh token을 발급한다. 현재 Auth API는 없다 |
| 자동로그인 | 결정 완료·구현 예정 | refresh token 만료 기간 차이로 처리한다 |
| 로그아웃 | 결정 완료·구현 예정 | refresh token을 폐기한다 |
| 회원 탈퇴 | 정책 일부 결정·구현 예정 | `WITHDRAWN` 상태를 사용하되 주문 이력 보존과 개인정보 삭제 범위는 별도 결정한다 |
| 인증 버전 | 도메인 구현·토큰 검증 미구현 | 상태 변경 시 증가한다. 기존 access token 거부 흐름은 후속 구현이다 |
| 로그인 실패 제한 | 도메인 구현·로그인 연결 미구현 | 5회 실패 시 `LOCKED`로 전환하는 규칙은 있다. 로그인 UseCase, rate limit, 감사 로그는 후속 구현이다 |
| 비밀번호 변경 | 보류 | 기존 비밀번호 확인과 새 비밀번호 검증 정책이 필요하다 |
| 비밀번호 재설정 | 보류 | 이메일 인증, 재설정 토큰, 만료 정책이 필요하다 |
| 이메일 변경 | 보류 | 새 이메일 중복 확인과 본인 확인 정책이 필요하다 |
| 닉네임 변경 API | 보류 | 현재 도메인 규칙만 있고 외부 기능은 없다 |
| 관리자 권한 부여 | 보류 | 누가 어떤 절차로 관리자 권한을 부여할지 정해야 한다 |
| 등급 변경 API | 보류 | 현재 도메인 규칙만 있고 외부 기능은 없다 |
| 회원 등급 자동 산정 | 보류 | 누적 구매액·주문 이력 기반 승급/강등 정책이 필요하다. 현재 등급은 정적 enum이며 수동 지정 전제다 |
| 계정 상태 | 도메인·JPA 구현 | `ACTIVE`, `LOCKED`, `SUSPENDED`, `WITHDRAWN`을 사용한다. 상태 변경 외부 API는 아직 없다 |
| 잠금 해제 | 보류 | 관리자 해제 또는 비밀번호 재설정 성공 중 결정이 필요하다 |

## 13. 구현 참고

이 문서는 비즈니스 정책을 설명한다.
현재 구현은 회원가입 API, 회원 도메인 규칙, Spring Security 기반 API 보호 경계를 중심으로 한다.

비밀번호 해시는 `PasswordHasher` 포트를 통해 처리하며, 현재 구현은 BCrypt를 사용한다.
