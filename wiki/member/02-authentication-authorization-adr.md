# ADR: 회원 인증·인가 방식 결정

작성: 2026-07-04 · 상태: 채택

구현 순서와 검증 계획은 [`03-authentication-implementation-plan.md`](./03-authentication-implementation-plan.md)를 따른다.

## 1. 배경

현재 회원 도메인은 회원가입, 비밀번호 해시, 회원 권한의 기본 의미를 제공한다.
다만 로그인, 로그아웃, 자동로그인, 회원 탈퇴·정지 상태, 인증된 사용자 기준의 API 접근 제어는 아직 구현되어 있지 않다.

현재 여러 API는 `memberId`를 요청 파라미터나 바디로 받는다.
인증이 도입되면 클라이언트가 전달한 `memberId`를 신뢰하지 않고, 서버가 인증 정보에서 회원 ID를 꺼내야 한다.

## 2. 사용자 질문 정리

이번 결정은 다음 질문에서 출발했다.

- 회원 인증에 어떤 기술을 사용할 것인가?
- 어떤 인증 방식을 차용하는 것이 좋은가?
- 비밀번호는 반드시 암호화되어야 한다.
- 자동로그인 기능이 추가될 수 있다.
- 로그인은 이메일과 비밀번호를 이용할 수 있다.
- 자동로그인은 어떤 방식으로 구현하는 것이 좋은가?
- 서버 세션, JWT 단독, Access JWT + Refresh Token, 별도 Remember-Me 방식은 무엇이 다른가?
- 회원 탈퇴·정지 상태가 필요한가?
- 로그인 실패 횟수 5회 도달 시 로그인이 불가능한 상태로 만들어야 한다.
- 운영 수준의 프로덕션을 목표로 할 때 단순안을 강화해야 하는가?

## 3. 결정

회원 인증은 **Spring Security + Access JWT + Refresh Token Rotation** 방식으로 구현한다.

자동로그인은 별도 remember-me 토큰을 만들지 않고, 로그인 요청의 `rememberMe` 값에 따라 refresh token 만료 기간을 다르게 두는 방식으로 처리한다.

```text
rememberMe=false
- access token: 5~10분
- refresh token: 12시간~7일

rememberMe=true
- access token: 5~10분
- refresh token: 14~30일
```

토큰 정책은 다음과 같다.

- access token은 JWT로 발급한다.
- refresh token은 랜덤 opaque token으로 발급한다.
- refresh token 원문은 저장하지 않고 해시값만 저장한다.
- refresh token 해시는 서버 비밀값을 섞은 HMAC 또는 pepper 기반 해시로 저장한다.
- refresh token은 재발급 시마다 회전한다.
- 이미 회전된 refresh token이 다시 사용되면 탈취 가능성으로 보고 같은 token family의 세션들을 폐기한다.
- 로그아웃 시 refresh token을 폐기한다.
- 전체 로그아웃이 필요하면 회원의 모든 refresh token을 폐기한다.
- 회원 상태 변경, 비밀번호 변경, 권한 변경, 전체 로그아웃 시 회원의 `authVersion`을 증가시켜 기존 access token을 무효화한다.

JWT claim은 최소 정보만 포함한다.

| claim | 의미 |
|---|---|
| `sub` | 회원 ID |
| `role` | 회원 권한 (`USER`, `ADMIN`) |
| `authVersion` | 발급 시점의 회원 인증 버전 |
| `sid` | 로그인 세션 ID |
| `iat` | 발급 시각 |
| `exp` | 만료 시각 |
| `jti` | 토큰 ID |

민감 정보, 비밀번호 해시, 주소, 전화번호 등은 JWT에 포함하지 않는다.

## 4. 비밀번호 정책

비밀번호는 평문으로 저장하지 않는다.

- 회원가입 시 평문 비밀번호를 비밀번호 해시값으로 변환해 저장한다.
- 로그인 시 입력 비밀번호와 저장된 해시값을 비교한다.
- 현재 구현의 `PasswordHasher` 포트와 BCrypt 구현을 유지하되, cost는 설정으로 관리한다.
- 향후 필요하면 `DelegatingPasswordEncoder` 기반으로 알고리즘 전환 여지를 둔다.
- 비밀번호 해시 알고리즘 전환을 위해 저장값에는 알고리즘 식별자를 포함할 수 있어야 한다.

문서와 코드에서 "암호화"라고 부르더라도 구현 의미는 복호화 가능한 암호화가 아니라 **단방향 비밀번호 해시**다.

## 5. 로그인 실패 잠금 정책

이메일과 비밀번호로 로그인한다.

로그인 실패 정책은 다음과 같다.

- 로그인 실패 횟수는 회원별로 누적한다.
- 비밀번호 불일치로 로그인에 실패하면 실패 횟수를 1 증가시킨다.
- 실패 횟수가 5회에 도달하면 계정을 로그인 불가능 상태로 전환한다.
- 잠금 상태의 회원은 올바른 비밀번호를 입력해도 로그인할 수 없다.
- 로그인 성공 시 실패 횟수는 0으로 초기화한다.
- 존재하지 않는 이메일과 비밀번호 불일치 응답은 외부에 구분해 노출하지 않는다.
- 로그인 실패는 계정별 카운트와 별개로 IP·이메일 기준 rate limit을 적용한다.
- 계정 잠금 시 refresh token을 폐기하고 `authVersion`을 증가시킨다.
- 로그인 성공, 실패, 잠금, 잠금 해제는 감사 로그로 남긴다.

잠금 상태 표현은 `MemberStatus`에 `LOCKED`를 추가해 표현한다.

```text
ACTIVE -> LOCKED
```

잠금 해제 방식은 별도 정책으로 둔다.
초기 구현은 관리자 해제 또는 비밀번호 재설정 성공 시 해제를 검토한다.

## 6. 회원 상태 정책

회원은 상태를 가진다.

| 상태 | 로그인 | refresh 재발급 | 주요 의미 |
|---|---:|---:|---|
| `ACTIVE` | 가능 | 가능 | 정상 회원 |
| `LOCKED` | 불가 | 불가 | 로그인 실패 5회로 잠긴 회원 |
| `SUSPENDED` | 불가 | 불가 | 운영 정책으로 정지된 회원 |
| `WITHDRAWN` | 불가 | 불가 | 탈퇴한 회원 |

상태 전이의 기본 방향은 다음과 같다.

```text
ACTIVE -> LOCKED
ACTIVE -> SUSPENDED
ACTIVE -> WITHDRAWN
LOCKED -> ACTIVE
SUSPENDED -> ACTIVE
```

`WITHDRAWN`은 원칙적으로 되돌리지 않는다.
탈퇴 회원의 주문 이력 보존, 개인정보 마스킹, 동일 이메일 재가입 허용 여부는 별도 탈퇴 정책에서 결정한다.

계정이 `LOCKED`, `SUSPENDED`, `WITHDRAWN` 상태가 되면 해당 회원의 refresh token은 폐기한다.
또한 `authVersion`을 증가시켜 이미 발급된 access token도 더 이상 통과하지 못하게 한다.

## 7. 인가 정책

인가 정책은 두 층으로 나눈다.

### 역할 기반 인가

관리 API는 `ADMIN` 권한만 접근할 수 있다.

예:

- 상품 등록·수정
- 브랜드 등록·수정
- 쿠폰 정책 생성
- 운영자 회원 상태 변경

### 소유권 기반 인가

회원 본인의 리소스는 인증된 회원 ID 기준으로 접근한다.

예:

- 내 카트 조회·변경·체크아웃
- 내 주문 조회·취소
- 내 회원 정보 조회·수정

인증 도입 후에는 클라이언트가 `memberId`를 요청 바디나 쿼리로 넘기는 방식을 점진적으로 제거한다.
Controller는 `SecurityContext`에서 인증된 회원 ID를 읽어 application command로 전달한다.
domain 계층은 Spring Security를 알지 않는다.

## 8. 운영 기본안: 인증 상태 확인 시점

운영 수준에서는 정지·탈퇴·권한 변경을 access token 만료까지 방치하지 않는다.
그렇다고 매 요청마다 DB를 조회하지도 않는다.

초기 운영 구현은 다음 방식을 따른다.

- access token 만료 시간을 5~10분으로 짧게 둔다.
- 모든 인증 요청에서 JWT 서명과 만료를 검증한 뒤 회원 인증 스냅샷을 확인한다.
- 회원 인증 스냅샷은 Redis 같은 캐시에 저장한다.
- 스냅샷에는 `memberId`, `status`, `role`, `authVersion`을 포함한다.
- 캐시가 없으면 DB에서 조회한 뒤 짧은 TTL로 캐시에 적재한다.
- JWT의 `authVersion`과 현재 회원의 `authVersion`이 다르면 인증을 거부한다.
- 현재 회원 상태가 `ACTIVE`가 아니면 인증을 거부한다.
- 권한 변경, 상태 변경, 잠금, 탈퇴, 전체 로그아웃 시 캐시를 무효화하고 `authVersion`을 증가시킨다.
- refresh token 재발급 시 회원 상태를 확인한다.
- 로그인 시 회원 상태를 확인한다.
- 관리자·결제·주문·회원 상태 변경처럼 민감한 API에서는 DB 기준 재확인을 허용한다.

이 방식은 JWT의 확장성 장점과 운영상 필요한 즉시 차단 요구 사이의 균형을 잡는다.
순수 stateless JWT보다 구현은 복잡하지만, 정지·탈퇴·권한 변경이 실제 서비스에서 즉시 반영되어야 하므로 운영 기본안으로 채택한다.

단일 기기 로그아웃은 해당 refresh session을 폐기한다.
access token은 짧은 만료 시간으로 제한하되, 즉시 차단이 필요한 경우 `sid` 또는 `jti` denylist를 access token 만료 시각까지 캐시에 저장한다.

## 9. 대안 비교

### 대안 A: 서버 세션

로그인 시 서버가 세션을 만들고 클라이언트는 세션 쿠키를 들고 다닌다.

장점:

- 서버에서 세션을 삭제하면 즉시 로그아웃된다.
- 브라우저 기반 서비스에서는 구현이 단순하다.
- 인증 상태를 서버가 강하게 통제할 수 있다.

단점:

- 모바일·외부 API·SPA 확장에서 불편할 수 있다.
- 다중 서버 환경에서는 Redis 같은 공유 세션 저장소가 필요하다.
- 쿠키 기반이므로 CSRF 정책을 함께 신경 써야 한다.

이번 결정에서는 채택하지 않는다.
현재 서비스는 REST API 백엔드이고 모바일·SPA 확장 가능성을 열어두는 편이 낫다.

### 대안 B: JWT Access Token 단독

로그인 시 긴 만료 시간을 가진 JWT 하나만 발급한다.

장점:

- 구현이 단순하다.
- 서버 저장소 없이 인증을 검증할 수 있다.

단점:

- 로그아웃과 강제 차단이 어렵다.
- 토큰이 탈취되면 만료 전까지 계속 유효하다.
- 자동로그인을 위해 만료 시간을 길게 잡으면 위험하다.
- 회원 정지·탈퇴 상태가 즉시 반영되지 않는다.

이번 결정에서는 채택하지 않는다.
자동로그인과 회원 정지·탈퇴 요구가 있으므로 운영 위험이 크다.

### 대안 C: Access JWT + Refresh Token Rotation

짧은 access token과 긴 refresh token을 함께 사용한다.

장점:

- API 요청은 JWT로 stateless하게 처리할 수 있다.
- access token 탈취 피해 시간을 짧게 제한할 수 있다.
- refresh token 저장소로 로그아웃, 전체 로그아웃, 기기별 세션 폐기가 가능하다.
- 자동로그인을 refresh token 만료 기간 차이로 표현할 수 있다.
- 회원 정지·탈퇴 상태와 권한 변경을 `authVersion`과 인증 스냅샷으로 빠르게 반영할 수 있다.

단점:

- refresh token 저장소가 필요하다.
- 토큰 회전과 재사용 감지 로직이 필요하다.
- 회원 인증 스냅샷 캐시와 무효화 정책이 필요하다.
- 프론트엔드가 access token 만료와 refresh 흐름을 처리해야 한다.

이번 결정에서 채택한다.
현재 요구사항의 단순함과 운영상 필요한 통제 사이에서 균형이 가장 좋다.

### 대안 D: Access JWT + 별도 Remember-Me Token

일반 refresh token과 자동로그인 remember-me token을 분리한다.

장점:

- 일반 로그인 유지와 자동로그인을 정책적으로 분리할 수 있다.
- 자동로그인 토큰에 별도 위험 기반 정책을 붙이기 쉽다.

단점:

- 토큰 종류가 늘어난다.
- 저장소, 검증, 폐기, 재발급 흐름이 복잡해진다.
- 현재 요구사항에는 과하다.

이번 결정에서는 채택하지 않는다.
자동로그인은 refresh token 만료 기간 차이로 충분히 표현할 수 있다.

## 10. 영향

구현 시 다음 변경이 필요하다.

- `MemberStatus` 추가
- 회원 로그인 실패 횟수 필드 추가
- 회원 `authVersion` 필드 추가
- 실패 5회 도달 시 `LOCKED` 전환
- 로그인 성공 시 실패 횟수 초기화
- refresh token 저장소 추가
- refresh token family와 session ID 저장
- 회원 인증 스냅샷 캐시 추가
- 상태·권한·비밀번호 변경 시 `authVersion` 증가와 캐시 무효화
- 로그인, refresh, logout, me API 추가
- 기존 memberId 입력 API를 인증된 회원 ID 기반으로 전환
- 관리자 API에 역할 기반 인가 적용
- 내 리소스 API에 소유권 기반 인가 적용
- 인증·인가 감사 로그 추가
- 로그인 실패 rate limit 추가
