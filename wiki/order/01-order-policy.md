# 주문 정책

작성: 2026-06-13 · 상태: 구현 완료(이번 이터레이션 범위)

이 문서는 주문이 비즈니스 관점에서 어떤 의미를 갖고, 어떤 규칙에 따라 생성·결제·취소되는지를 정의한다.
구현 구조와 기술적 근거는 [`order-domain-spec.md`](./order-domain-spec.md)를 따른다.

## 1. 목적

주문 정책은 고객의 구매 의사가 시스템 안에서 언제 주문으로 인정되는지, 결제 결과와 취소가 주문·재고·환불에 어떤 영향을 주는지를 정의한다.

이 정책은 다음 질문에 답한다.

- 고객의 주문 요청은 언제 유효한 주문이 되는가?
- 주문 총액과 쿠폰 할인 이후 청구액은 무엇을 기준으로 확정되는가?
- 결제 성공과 실패는 주문 상태를 어떻게 바꾸는가?
- 주문 취소 시 무엇을 되돌려야 하는가?
- 주문 내역은 상품 변경 이후에도 어떤 모습으로 보여야 하는가?

## 2. 핵심 용어

| 용어 | 의미 |
|---|---|
| 주문 | 고객이 하나 이상의 판매 단위를 구매하겠다는 의사를 시스템에 기록한 것 |
| 주문 항목 | 주문에 포함된 개별 판매 단위와 수량 |
| 판매 단위 | 고객이 실제로 구매하는 SKU |
| 주문 총액 | 주문 항목별 주문 시점 판매가와 수량을 곱해 합산한 할인 전 금액 |
| 할인액 | 쿠폰으로 차감된 금액 |
| 청구액 | 주문 총액에서 할인액을 뺀 실제 결제·환불 기준 금액 |
| 결제 대기 | 주문은 생성되었고 재고는 차감되었지만 결제 결과가 아직 확정되지 않은 상태 |
| 결제 완료 | 결제가 성공하여 주문이 유효한 구매 사실로 확정된 상태 |
| 취소 | 주문을 무효화하고, 필요한 재고 복원과 환불을 수행하는 처리 |
| 재고 복원 | 주문 실패나 취소로 인해 차감했던 판매 가능 재고를 되돌리는 것 |
| 쿠폰 복원 | 주문 실패나 취소로 인해 사용 처리했던 쿠폰을 미사용 상태로 되돌리는 것 |

## 3. 주문 성립 규칙

주문은 고객, 주문 항목, 구매 가능한 상품, 충분한 재고가 모두 확인되었을 때 생성된다.

- 주문자는 필수다.
- 주문은 최소 하나 이상의 주문 항목을 가져야 한다.
- 각 주문 항목은 SKU ID를 가져야 한다.
- 각 주문 항목의 수량은 1개 이상이어야 한다.
- 존재하지 않는 고객의 주문은 생성할 수 없다.
- 존재하지 않는 상품이나 SKU는 주문할 수 없다.
- 판매 중이 아닌 상품은 주문할 수 없다.
- 비활성 브랜드의 상품은 주문할 수 없다.
- 판매 가능 재고가 부족한 SKU는 주문할 수 없다.
- 주문 생성 시점에 주문 항목의 상품 정보, 옵션 정보, 판매가를 기록한다.

주문 생성은 고객의 구매 요청을 기록하는 행위이지만, 결제 성공 전까지 최종 구매 완료를 의미하지 않는다.

## 4. 주문 금액 정책

주문 총액은 주문 생성 시점의 SKU 판매가를 기준으로 확정한다.

- 주문 항목 금액은 `주문 시점 판매가 × 수량`이다.
- 주문 총액은 모든 주문 항목 금액의 합이다.
- 쿠폰을 사용하지 않으면 할인액은 0원이고 청구액은 주문 총액과 같다.
- 쿠폰을 사용하면 할인액을 주문 총액에서 차감하고, 청구액을 별도로 확정한다.
- 주문 단가는 필수다.
- 주문 총액, 할인액, 청구액은 모두 필수다.
- 상품 가격이 나중에 변경되어도 이미 생성된 주문 총액, 할인액, 청구액은 변경되지 않는다.
- 주문 총액은 외부 입력값으로 신뢰하지 않고, 주문 항목을 기준으로 계산한다.
- 쿠폰 할인액은 주문 생성 시점의 쿠폰 규칙으로 계산한다.
- 청구액은 `주문 총액 - 할인액`이며 결제와 환불의 기준이다.
- 주문 총액, 할인액, 청구액은 모두 주문의 사실 기록으로 보존한다.

쿠폰이 없던 시기에는 주문 총액과 청구액이 같았지만, 쿠폰 도입 이후 두 금액은 분리된다.
고객에게 할인 전 금액과 실제 청구액을 모두 설명할 수 있어야 하므로 세 금액을 함께 보존한다.

## 5. 주문 상품 스냅샷 정책

주문은 과거 구매 사실의 기록이다. 따라서 주문 내역은 현재 상품 정보가 아니라 주문 당시 정보를 보여줘야 한다.

주문 항목에는 다음 정보를 주문 시점 기준으로 보존한다.

- 상품 ID
- SKU ID
- 상품명
- 옵션 요약
- 주문 시점 판매가
- 수량
- 주문 항목 금액

이 정책의 의도는 다음과 같다.

- 고객은 자신이 구매한 당시의 상품명, 옵션, 가격을 확인할 수 있어야 한다.
- 상품명이 변경되어도 과거 주문 내역의 상품명은 바뀌지 않는다.
- 상품이 판매 중지되거나 단종되어도 과거 주문 내역은 유지된다.
- 브랜드가 비활성화되어도 이미 생성된 과거 주문 내역은 유지된다.
- 주문 조회는 현재 상품 상태에 영향을 받지 않아야 한다.

주문 항목 스냅샷은 다음 규칙을 따른다.

- 상품 ID와 SKU ID는 필수다.
- 상품명은 비어 있을 수 없다.
- 옵션 요약은 비어 있을 수 없다.
- 옵션 요약은 현재 `옵션명:옵션값 / 옵션명:옵션값` 형식으로 기록한다.

## 6. 주문 상태 정책

주문은 다음 상태를 가진다.

| 상태 | 비즈니스 의미 | 가능한 전이 |
|---|---|---|
| 결제 대기 | 주문이 생성되고 재고가 차감되었으며 결제 결과를 기다리는 상태 | 결제 완료, 취소 |
| 결제 완료 | 결제가 성공하여 고객의 구매가 확정된 상태 | 취소 |
| 취소 | 주문이 무효화된 최종 상태 | 없음 |

취소 상태는 결제 실패로 인한 시스템 취소와 고객 요청으로 인한 취소를 모두 포함한다.
현재 정책에서는 두 경우 모두 결과가 "주문 무효 + 재고 복원"으로 같기 때문에 별도 상태로 나누지 않는다.

취소 사유 구분이 필요해지면 취소 사유를 별도 정책으로 추가한다.

## 7. 결제 정책

결제는 주문 생성 이후 수행된다.

- 결제 성공 시 주문은 결제 완료 상태가 된다.
- 결제 실패 시 주문은 취소 상태가 된다.
- 결제 실패로 취소된 주문도 삭제하지 않고 기록으로 남긴다.
- 결제 실패 시 주문 생성 과정에서 차감한 재고를 복원한다.
- 쿠폰 적용 주문의 결제가 실패하면 사용 처리했던 쿠폰도 복원한다.
- 결제 실패 주문은 결제가 완료되지 않았으므로 환불하지 않는다.
- 청구액이 0원이면 결제 게이트웨이를 호출하지 않고 결제 완료로 처리한다.

결제 실패 주문을 삭제하지 않는 이유는 다음과 같다.

- 고객의 주문 시도와 실패 이력을 추적할 수 있어야 한다.
- 운영·감사 관점에서 주문 실패의 원인을 확인할 수 있어야 한다.
- 재고 복원 등 보상 처리가 실제로 수행되었는지 확인할 수 있어야 한다.

## 8. 취소 정책

결제 대기 주문과 결제 완료 주문은 취소할 수 있다.

취소가 완료되면 다음 처리가 함께 이루어진다.

- 주문 상태를 취소로 변경한다.
- 주문 항목별로 차감했던 판매 가능 재고를 복원한다.
- 쿠폰 적용 주문이면 사용 처리했던 쿠폰을 복원한다.
- 결제 완료 상태였던 주문은 청구액에 대한 환불을 요청한다.
- 결제 대기 상태였던 주문은 환불하지 않는다.

취소된 주문은 다시 결제 완료 상태로 돌아갈 수 없다.
이미 취소된 주문은 다시 취소할 수 없다.

현재 정책에서는 주문 전체 취소만 다룬다. 부분 취소, 부분 반품, 환불 실패 처리는 별도 정책으로 확장한다.
부분 취소가 도입되면 쿠폰 할인 배분과 쿠폰 부분 복원 여부를 함께 정해야 한다.

## 9. 재고 정책

주문은 판매 가능 재고에 직접 영향을 준다.

- 주문 생성 시 주문 항목 수량만큼 SKU의 판매 가능 재고를 차감한다.
- 결제 실패 시 차감했던 판매 가능 재고를 복원한다.
- 주문 취소 시 차감했던 판매 가능 재고를 복원한다.
- 재고 부족 상태에서는 주문을 생성할 수 없다.
- 재고 차감 중 충돌이 감지되면 주문 요청은 실패하며, 호출자가 다시 시도해야 한다.
- 현재 지원하는 재고 차감 방식은 낙관적 방식, 비관적 방식, 조건부 원자 차감 방식이다.

재고는 고객에게 판매 가능한 수량을 의미한다.
실물 창고 재고와의 관계는 상품 정책과 별도 재고 연동 정책에서 다룬다.

## 10. 조회 정책

주문은 단건 조회와 회원별 목록 조회를 제공한다.

- 존재하지 않는 주문을 조회하면 찾을 수 없는 주문으로 처리한다.
- 회원별 주문 목록은 인증된 회원 ID를 기준으로 조회한다.
- 주문 상세 조회와 취소는 인증된 회원 본인의 주문에만 허용한다.
- 다른 회원의 주문 ID로 상세 조회나 취소를 요청하면 현재 ErrorType 정책에 맞춰 `NOT_FOUND`로 응답한다.
  이는 주문 존재 여부를 외부에 노출하지 않기 위한 선택이다.
- 조회 응답에는 주문 상태, 주문 항목, 주문 항목 금액, 주문 총액, 할인액, 청구액, 사용 쿠폰 ID를 포함한다.

## 11. 예외와 보류 정책

현재 정책에서 의식적으로 보류한 항목은 다음과 같다.

| 항목 | 현재 결정 | 향후 필요성 |
|---|---|---|
| 멱등성 | 보류 | 같은 주문 요청이 반복될 때 이중 주문·이중 결제를 막아야 한다 |
| 환불 실패 처리 | 보류 | 실제 결제 연동 시 재시도·수동 개입 정책이 필요하다 |
| 인증·인가 | 적용 | 인증 회원 본인의 주문만 조회·취소할 수 있다 |
| 부분 취소 | 보류 | 주문 항목 단위 취소, 환불 금액 계산, 쿠폰 할인 배분과 부분 복원 정책이 필요하다 |
| 부분 반품 | 보류 | 배송 이후 반품 가능 여부와 재고 복귀 기준이 필요하다 |
| 주문번호 | 보류 | 고객·운영자가 식별하기 쉬운 외부 노출 번호가 필요할 수 있다 |
| 동일 SKU 중복 라인 | 해소 | 장바구니 정책에서 같은 SKU는 수량 합산으로 결정했다. 주문 요청 라인 병합·거부 정책이 필요하면 주문 입력 검증으로 별도 정의한다 |

## 12. 구현 참고

이 문서는 비즈니스 정책을 설명한다.
구현 방식, Aggregate 구조, 트랜잭션 경계, 재고 차감 동시성 전략, 계층 매핑은 [`order-domain-spec.md`](./order-domain-spec.md)를 기준으로 한다.

현재 기본 결제 게이트웨이는 지연을 흉내 내고 항상 성공하는 스텁이다.
결제 실패 보상 흐름은 실패하는 결제 게이트웨이 대역을 통해 검증한다.
